PMIS AI — Contract-First Architecture Refactor
P0 / P1 Product & Engineering Backlog

0. Mission
This repository already contains a broad Taiwan public-works PMIS covering PCCES BOQ, daily site logs,
valuations, payments, cost, progress, quality, safety, ITP, submittals, RFIs, acceptance, portfolio
management, and role-based workflows.


Do not add more horizontal PMIS modules at this stage.


The product north star is:


       Upload the contract, specifications, quality plan, and required forms.
       AI converts those documents into traceable execution requirements, and the system tells
       each project participant what must be done, when, why, what evidence is required, and
       which contractual source created the obligation.


The current product is predominantly BOQ-first and module-first.


The target architecture is Contract-First × Requirement-Driven, while retaining the existing PCCES BOQ
as the financial/progress spine.


The final product should have two connected structural spines:



  FINANCIAL / PROGRESS SPINE


  PCCES BOQ
     ↓
  Work Item
     ↓
  Daily Quantity
     ↓
  Valuation
     ↓
  Payment / Progress / Cost



  EXECUTION / COMPLIANCE SPINE


  Contract / Specification / Quality Plan
     ↓
  Requirement
     ↓
  Work Item / Project Phase / Event
      ↓



                                                   1

     ↓
  Execution Record
     ↓
  Inspection / Defect / Report / Audit


These two spines must be linked through work_item_id and project workflow events where
applicable.

P0-03 — Separate System Administration from Contractual
Authority

Problem

The current project admin override allows an admin to obtain approval authority.


Project administration and contractual approval authority are different concepts.


A contractor project creator must never become authorized to:


    • approve their own valuation
    • judge their own inspection as passed
    • close their own defect
    • approve their own submittal

simply because they created the project.


Rule


  TECHNICAL ADMINISTRATION
  ≠
  CONTRACTUAL AUTHORITY



Implement

is_project_admin may authorize:


    • member management
    • project settings
    • integrations
    • project deletion
    • technical configuration

It must not automatically authorize workflow transitions.


Workflow authorization must derive from project party and project role.




                                                    7
Create explicit permission helper functions or equivalent server-side authorization functions.


Examples:



  can_submit_inspection()
  can_decide_inspection()
  can_submit_valuation()
  can_approve_valuation()
  can_close_defect()
  can_review_submittal()
  can_ratify_change_order()


Do not use one generic can_write() helper for sensitive workflow authority.


RLS protects row access.


Database triggers or RPC boundaries protect workflow state transitions.


Acceptance Criteria

Test the following negative cases at DB level:



  Contractor admin → cannot approve own valuation
  Contractor admin → cannot pass own inspection
  Contractor admin → cannot close own defect
  Contractor admin → cannot approve own submittal


  Owner admin → cannot perform supervisor inspection judgment


  Supervisor → can decide inspection
  Supervisor → can review submittal
  Supervisor → can close defect


  Authorized owner role → can perform defined owner approval action


Frontend permission behavior must match server behavior.


No frontend-only authorization.
