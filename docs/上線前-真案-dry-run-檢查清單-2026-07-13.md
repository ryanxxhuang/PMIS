# 上線前:真實案 Dry-Run 檢查清單

目的：在**正式站**用一個**真實專案 + 真實 PCCES 標單**，以三種角色、桌機＋手機各跑一輪完整週期。
這是找出「只在真登入＋真資料才會冒出來」問題的最有效方法（剛那輪 Codex 報告的嚴重 bug 都是這樣抓到的）。

- 主站：`https://ryanxxhuang.github.io/PMIS/`
- 每一步：做完 → 打勾；不如預期 → 記在最後「問題紀錄」（附頁面／角色／資料狀態／截圖）。
- **特別盯**標「🔴 剛修」的項目——這些是上一輪剛修好、還沒獨立複測的，要確認在真環境真的解掉。

## 準備
- [ ] 三個真帳號：施工廠商 / 監造 / 機關（或至少一個 admin 帳號能切角色）。
- [ ] 一份**真實 PCCES 標單 XML**（幾千項那種，不要用範例）。
- [ ] 一份**真實契約 PDF**、一份**施工計畫 PDF**（送審用）。
- [ ] 幾張**真實工地照片**（含至少一張混凝土澆置）＋一張**非工地照**（測 AI 判非工地）。
- [ ] 手機一支（或瀏覽器 375px viewport）。

---

## 階段 0：建立專案
- [ ] 建立新專案。🔴 剛修 **P1-04**：所有欄位應**空白**、只有 placeholder 範例；施工廠商**不應**自動帶你的公司名。
- [ ] 填真實工程名稱、機關、廠商、監造、開工/竣工日，建立成功。

## 階段 1：上傳契約與標單
- [ ] 到「契約與文件」用**正式上傳鈕（file input）**上傳契約 PDF。🔴 剛修 **P0-08**：應**上傳成功**，不出現 `invalid input syntax for type uuid: "projects"`。
- [ ] 契約上傳後，系統應自動抽出「履約需求」與「義務時程」。到「履約需求」頁確認有東西。
- [ ] 到「標單工項」上傳真實 PCCES XML → 匯入成功、工項數與總價正確。🔴 剛修 **P1-11**：真實空案**不應**有「用範例標單試試」。
- [ ] 空案時各頁的空狀態（P1-05）：施工日誌、估驗應顯示「缺標單 + 前往匯入 CTA」，而非空白。

## 階段 2：施工日誌（含 AI）
- [ ] 以**施工廠商**填一天日誌：日期、天氣、工作摘要、幾個工項的當日數量，存檔成功、F5 保留。
- [ ] 設工地座標 → 點「帶入天氣」→ 應回中央氣象局上午/下午天氣。🔴 剛修 **P1-01**：改成**過去日期**再點，應顯示**友善中文訊息**（非「Edge Function returned a non-2xx」）。
- [ ] 「複製昨日」：新日期一鍵帶入前一筆班組/機具/材料。
- [ ] **AI 批次辨識照片**：一次選 3–5 張（含混凝土澆置照＋那張非工地照）。
  - [ ] 逐張出說明、分類、配工項。🔴 剛修 **P0-03**：非工地照應標「疑似非工地照」且**未自動配工項**；任何照片說明**不應**出現「完成/測試中/就位」等狀態詞。
  - [ ] 🔴 剛修 **P1-02**：每張可**搜尋改選/清除工項**；辨識失敗的張填說明後**能上傳**。
  - [ ] 覆核後全部上傳，照片簿顯示 AI 說明。

## 階段 3：估驗與請款（最該仔細看——牽涉錢）
- [ ] 建立估驗期，填幾個工項的累計數量（含一個小量的，如某工項 2 m³）。
- [ ] 「AI 估驗草擬」掃日誌帶入數量，覆核。
- [ ] **組請款佐證包**。🔴 剛修 **P0-01**：明細「本期完成數量」應等於你填的量（如 **2**，不是金額）；本期金額 = 量 × 單價（如 5,908，不是再乘一次的天文數字）；AI 施工說明的數量要對；頁首/合計/明細三者一致。
- [ ] 🔴 剛修 **P0-02**：佐證照片可**逐張 ✕ 排除**；排除後合計張數與 AI 說明不含它。
- [ ] 送監造審核 → 監造核定 → 請款/收款。🔴 剛修 **P1-07**：試著填「實收金額但不填收款日」或「負實收」→ 應被**擋下**；未收為負時應顯示紅色「實收超過應領，請查核」。
- [ ] 列印估驗單、列印請款佐證包（桌機）。

## 階段 4：品質與工安（含 AI）
- [ ] 以施工廠商：建查驗申請、開品質缺失、填自主檢查、澆置日建混凝土試體。
- [ ] 以監造：查驗判定、缺失複查結案。
- [ ] **工安**：開工安缺失 → 拍/選那張混凝土澆置照 → AI 判讀。🔴 剛修 **P0-04**：改善建議**不應有任何具體尺寸/距離數字**（不再自補 1.1m/1.5m），應寫「依適用條文與現場量測辦理」；不捏造法條條號。

## 階段 5：送審與 RFI（含 AI）
- [ ] 以施工廠商：提送一份「施工計畫」送審，**上傳施工計畫 PDF**。
- [ ] 以監造：**AI 讀文件審查**。🔴 剛修 **P0-05**：若契約需求裡有罰則/月報/竣工文件期限，這些應標**「不適用」**且**不因它們建議退回**；文件沒寫的應誠實標「未涵蓋」不假裝符合。🔴 剛修 **P1-10**：應出現「下載→抽字→比對，約 20–30 秒」狀態。
- [ ] 「AI 審查助手」出審查要點；帶入意見後核准/核備。🔴 剛修 **P2-03**：caution 不應出現 literal `\n`。
- [ ] 以施工廠商：提一筆 RFI（含設計判斷型，如「樑柱鋼筋與機電套管衝突該如何調整」）。
- [ ] 以監造：**AI 回覆草稿**。設計判斷型應說「建議轉設計單位釋疑」，**不硬編答案**。

## 階段 6：契約義務、罰款、變更、驗收
- [ ] 「契約與文件 → 義務時程」：義務依基準日算到期、逾期倒數、罰則顯示。
- [ ] 🔴 剛修 **P2-06**：若有逾期且罰則含罰率（千分之X / N‰ / N 元/日），應顯示「預估逾期違約金約 NT$X」；抽不出格式時顯示「格式需人工確認」而非消失。
- [ ] 變更設計：提出→審核→核准流程；驗收結算逐階段。

## 階段 7：機關視角
- [ ] 以**機關**登入。🔴 剛修 **P1-06**：看施工日誌/品質應有「機關監督檢視…唯讀」橫幅、輸入呈 disabled、文案不再寫「監造帳號」。🔴 剛修 **P1-12**：BOQ **不應**有「重新匯入標單」。
- [ ] 「風險稽核 → 文件勾稽鏈」：應抓到真實不一致（估驗 vs 日誌、澆置無試體等）。
- [ ] 「AI 稽核意見」。🔴 剛修 **P0-06**：意見/建議**不應**出現剔除/補強/停工/罰款等處置決策詞，只建議查證。

## 階段 8：多專案 / 切案
- [ ] 再建一個全新空案，從有資料案**切過去**。🔴 剛修 **P0-07**：切過去後（不刷新）Portfolio/提醒中心**不應**看到前案的缺失/付款。

## 階段 9：手機（375px）全流程
- [ ] 手機跑一次：填日誌、拍照上傳、看估驗、開工安缺失。
- [ ] 🔴 剛修 **P1-08**：請款佐證包在手機**整頁不應水平漂移**（只表格容器自身可橫捲）。
- [ ] 🔴 剛修 **P2-04**：頂列專案名 hover/長按應可見全名。
- [ ] RFI 長內容應截斷（P1-09）；重要按鈕點得到。

## 階段 10：AI Copilot（右下角浮動鈕）
- [ ] 問跨模組彙整題（「本案進度、金流、品質現況？」）→ grounded 回答＋出處。
- [ ] 問特定義務（「月報什麼時候要交？」）→ 🔴 剛修 **P1-03**：應能引用該義務（含契約出處）。
- [ ] 🔴 剛修 **P2-02**：回答不應有 literal `**` 星號。
- [ ] （若你的金流資料剛好有矛盾）問金流 → 🔴 剛修 **P0-09**：應主動說「資料不一致、無法判定」，不硬答。
