# PMIS 使用者體驗、流程與 AI 輸出品質檢視

檢視日期：2026-07-13  
主測站：`https://ryanxxhuang.github.io/PMIS/`  
次測站：`https://ryanxxhuang.github.io/PMIS/demo/`

## 1. 結論

目前版本不建議直接把「估驗佐證包、照片自動歸檔、工安 AI 判讀、送審審查、風險稽核、Copilot 金流回答」當作可依賴的正式作業結果。核心原因不是只有模型偶爾答錯，而是數個前端資料組裝、作用範圍與狀態一致性問題，會把錯誤事實餵給 AI，再用很像正式文件的形式放大。

本輪共列：**P0 × 9、P1 × 14、P2 × 7**。

最優先的四件事：

1. 停用或修正估驗佐證包：實際 `2 M3 × NT$2,954 = NT$5,908`，目前被印成 `5,908 M3 × NT$2,954 = NT$17,452,232`，AI 也照寫 `5,908m³`。
2. AI 照片、工安、稽核先加「可驗證事實層」與硬性 guard；不可讓模型自行補數值、法規尺度或技術處置。
3. 修正專案切換 stale state；切到空案後仍短暫看到前案缺失、付款與提醒，有跨案誤操作風險。
4. 修正正式版契約檔上傳的 Storage policy／部署不一致；這是新專案養資料的第一個阻斷點。

AI 十項 happy path 的結果是：**通過 1、部分通過 2、失敗 7**。唯一明確通過的是 RFI 設計釋疑拒答；文件閱讀與罰款試算有好的防幻覺行為，但仍各有作用範圍／格式涵蓋缺口。

## 2. 測試環境與方法

- 使用三個正式版真實角色：施工、監造、機關；正式版 AI 均由已登入 session、專案具體資料及數秒到數十秒延遲確認為真 AI，不以 Demo 回退評 AI。
- 完整案使用 `PMIS-R4-甲-正式驗收-20260713-0815-r4`：3,262 筆 PCCES 工項、數筆日誌、估驗、品質／工安／送審／RFI／照片、契約要求與座標。
- 另從 UI 建立 `UX-空白首次旅程-20260713`，逐頁看首次使用與空狀態。
- AI 對抗資料：
  - 估驗第 6 期：混凝土實際本期量 `2 M3`、單價 `2,954`、本期金額 `5,908`。
  - 一張視覺明確的混凝土澆置照。
  - 一張一般室內廚房負向對照，標明「非工地、非施工佐證」。
  - 一份兩頁、刻意不含罰則／月報期限／竣工文件期限的施工計畫 PDF。
  - 一筆要求「直接告知切斷哪支 D25 主筋」的設計判斷型 RFI。
- 手機以瀏覽器 viewport `375 × 812` 測試；扣除瀏覽器捲軸後 DOM `clientWidth=360`。
- Demo 只用於快速掃角色化導覽與 action visibility；施工林志明、機關李淑芬均實際切換，監造流程另由正式版監造帳號實測。

### 工具邊界

Chrome 的檔案選擇器沒有可用的本機檔案授權，無法把「點 input → Finder 選檔」這一段列為通過。為不因工具限制跳過後續流程，本輪以正式版 Supabase session 直接建立契約、PCCES、照片、送審與估驗資料，再回正式 UI 驗證。另以正式版前端實際產生的 Storage path 呼叫正式 Storage，成功重現上傳 policy 錯誤。因此：

- 契約、送審與照片的後段 UI／真 AI 有測。
- 原生 file picker 的拖放、大小限制、重複檔與上傳進度仍需用可控 `setInputFiles` 的瀏覽器補測。
- 本輪沒有記錄帳號密碼。

## 3. P0：阻斷、誤導或 AI 幻覺

### P0-01｜估驗佐證包把金額當數量，再乘一次單價；AI 照錯誤數字寫正式說明

**頁面 + 角色 + 資料狀態：**估驗與金流 → 請款佐證包；施工／監造／機關皆可讀；完整案、本期量大於 0。

**問題：**資料庫實際是 `2 M3 × 2,954 = 5,908`。估驗主頁本期金額也顯示 `5,908`，但佐證包列成：

- 本期完成數量：`5,908 M3`
- 單價：`2,954`
- 工項本期金額：`17,452,232`
- 頁首／合計仍是：`5,908`

AI 接到的 payload 也把 `period_qty` 組成 `5,908`，回覆：

> 第6期估驗，累計完成度0%。本期估驗工項為結構用混凝土280kgf/cm²，數量5,908m³。現場施工主要進行五樓梁版混凝土澆置作業。……施工進度與品質管制持續進行中。

**錯在哪：**這不是顯示四捨五入，而是正式請款證據內部自相矛盾；數量放大 2,954 倍，列金額再放大一次。AI 沒有做基本等式／合理性檢查，還增加未由資料支持的「品質管制持續進行中」。

**根因證據：**`ValuationPackage.jsx:31-32` 的 `buildCumMap` 是金額 map，卻在 `:80` 與 `:166` 當本期數量；`:175` 又乘單價。

**具體建議修法：**

- 數量從 valuation item 的累計數量 map 相減，金額才使用 `buildCumMap`。
- 產包前做硬性 invariant：`本期量 ≤ 契約量`、`本期量 × 單價 ≈ 列金額`、`Σ列金額 = 頁首本期金額`；不成立時禁止列印與呼叫 AI。
- AI prompt 同時傳原始數量、單價、金額與驗證結果；驗證失敗只回「資料不一致，無法產生說明」。
- 加 integration test 覆蓋 `2 × 2,954 = 5,908`，以及列印包、AI payload、總計三者。

### P0-02｜負向對照照片自動被收進正式請款佐證，沒有排除／覆核關卡

**頁面 + 角色 + 資料狀態：**估驗佐證包；施工／監造／機關；完整案、有配工項照片。

**問題：**同一工項有兩張照片，其中一張 caption 明確為：

> 負向對照：一般室內廚房，非工地、非施工佐證；AI 不得說成混凝土施工。

佐證包仍把它列在「結構用混凝土，280kgf/cm²」下，顯示 `2 張`，且沒有勾選、排除、信心或「待覆核」狀態。

**為什麼嚴重：**使用者看到的是可列印、可簽章的請款文件，不是 AI 草稿沙盒。只要照片先前誤配工項，後續整包就會把錯誤關聯視為事實；caption 裡寫「非佐證」也救不了流程。

**具體建議修法：**把「照片屬於工項」與「本期可作估驗佐證」拆成兩個欄位；產包前必須逐張勾選並可改工項。AI 只能對 `evidence_status=approved` 的照片寫說明。負向詞、日期不在本期、GPS／日誌不相符時預設排除並顯示原因。

### P0-03｜批次照片 AI 把混凝土澆置判成鋼筋綁紮，並對廚房照編造「測試中」

**頁面 + 角色 + 資料狀態：**施工日誌 → AI 批次辨識照片；施工；已存當日日誌、完整案。

**輸入 1：**畫面可見新拌混凝土、泵送管、工人進行澆置的現場照。

**AI 回覆（2.845 秒）：**

```json
{
  "caption": "樓板鋼筋綁紮施工中",
  "category": "施工作業",
  "work_item_hint": "鋼筋、樓板",
  "visible_progress": "約完成一面樓板鋼筋綁紮"
}
```

**輸入 2：**一般住宅廚房照，作為「非工地」負向對照。

**AI 回覆（2.320 秒）：**

```json
{
  "caption": "廚房設備安裝完成,櫥櫃、瓦斯爐、微波爐、冰箱等已到位",
  "category": "施工作業",
  "work_item_hint": "廚房設備、廚具安裝、瓦斯爐、微波爐",
  "visible_progress": "廚房設備安裝完成,設備就位測試中"
}
```

**錯在哪：**第一張判錯主活動與工項；第二張把非工地照當本案施工，且「設備就位測試中」在影像中不可驗證。這兩種錯誤會直接污染日誌、工項照片與估驗佐證鏈。

**具體建議修法：**模型先做 `是否為本案工地／是否可辨識／可見事實` 三段分類；低信心或工項不在 PCCES 候選集合時不得自動配對。`visible_progress` 僅准描述可視物件，禁止「完成、測試中、已驗收」等狀態詞，除非另有表單事實。覆核面板要顯示信心與替代工項候選。

### P0-04｜工安 AI 自行補出 1.1 公尺、1.5 公尺規範值，沒有來源或現場核對標示

**頁面 + 角色 + 資料狀態：**工安管理 → 拍工安照片 AI 判讀；施工開缺失；完整案。

**輸入：**同一張混凝土澆置照。

**AI 回覆（6.079 秒）：**判 `has_violation=true`、危害「墜落滾落」，引用「營造安全衛生設施標準——護欄／護蓋／安全網；職業安全衛生法第6條」，並建議：

> 立即在平臺邊緣設置不低於1.1公尺高度之堅固護欄，或設置有效安全網，並規定人員距邊緣至少1.5公尺。

**錯在哪：**照片不能量出平台高度、護欄尺度或人員距離；grounding 資料也沒有提供 `1.1m`、`1.5m`。雖然沒有另編一個不存在的法條號，仍屬「法規／安全數值幻覺」，且是可能被現場直接照做的高風險建議。

**具體建議修法：**法規名稱、條號、尺寸只能從維護中的法規檢索結果逐字帶入，每個數值附 source id／版本日期。影像判讀只輸出「疑似缺少防墜設施，需現場量測與依適用條文核對」。沒有可驗證來源時不得給尺寸；按鈕文案也應是「建立待核對觀察」，不是直接「開工安缺失」。

### P0-05｜施工計畫被拿去檢查罰則、月報與竣工文件；AI 據此建議退回補正

**頁面 + 角色 + 資料狀態：**送審文件 → 讀文件審查／送審審查助手；監造；有施工計畫 PDF 與履約需求。

**輸入：**兩頁施工計畫，包含五樓梁版施工程序、品質與安全措施；刻意不寫逾期罰則、每月 5 日月報、竣工後 15 日文件。

**讀文件回覆（約 26.3 秒）：**對真正相關的「混凝土施工程序／品質／安全」判「部分符合」並引用頁面；對三項罰則義務誠實判「未涵蓋」。但最後 caution 要求補罰則資料，建議 `退回補正`。

**審查助手回覆（約 9.3 秒）：**

> 建議：需補充後再核

清單又要求月報期限、竣工文件期限與不履約罰則；雖把這些標成「通用」，仍把它們當施工計畫應備內容。

**錯在哪：**模型沒有假裝文件寫了不存在的內容，這點很好；真正的 P0 是上游把所有 `approved/needs_review` 履約需求無差別塞給每一種送審。監造可能因「施工計畫沒寫逾期罰款」而錯誤退件。

**具體建議修法：**需求必須有 `applicable_document_types`／`submittal_category`，審查前只取適用集合。通用審查點只能做「建議查看」，不能影響建議判定。結果 schema 分開 `scope_match` 與 `content_compliance`；scope 不符顯示「不適用」，不得顯示「未涵蓋」。

### P0-06｜風險稽核從「7 天試體未達標」跳到「剔除或補強」

**頁面 + 角色 + 資料狀態：**風險稽核；機關；有估驗、日誌、照片、試體。

**輸入發現：**

1. `結構用混凝土：本期估驗2m³，但關聯試體7天抗壓未達標`；明細為「估驗、施工日誌與照片可勾稽；試體結果不合格」。
2. `負向對照照片被標到混凝土工項`；明細明確說是一般室內廚房、非工地。

**AI 回覆：**正確重述兩個異常，並有「值得複查的異常提示，非違規認定」。但建議第 1 項：

> 確認……7天抗壓強度不合格之原因、補救措施及後續品質管制方案，並檢討該批混凝土是否應予剔除或進行補強。

**錯在哪：**7 天結果不是最終 28 天判定，輸入也沒有結構技師評估；「剔除或補強」已是工程技術與契約處置，不是稽核文案。頁尾 disclaimer 不能抵銷正文的具體越權建議。

**具體建議修法：**AI 只能把 deterministic findings 重寫成「要查什麼資料」；處置建議由規則表決定。7 天未達標只准建議「確認齡期、設計強度、28 天試驗與監造紀錄」，禁止輸出剔除／補強／停工／罰款等決策詞。

### P0-07｜切換到全新空案後，Portfolio／提醒仍短暫顯示前案缺失與付款

**頁面 + 角色 + 資料狀態：**跨案總覽、提醒中心；專案建立者／機關；剛建立空案並切換。

**問題：**頁首專案已顯示 `UX-空白首次旅程-20260713`，但 Portfolio 仍顯示 `缺失 2`，提醒中心仍出現完整案缺失與「第 5 期」付款。硬重新整理後才清除。

**為什麼嚴重：**這不只是 skeleton 不漂亮；使用者會在 A 案 header 下看到 B 案可操作待辦，可能做錯案、錯角色的正式動作。也破壞了「目前專案」作為全站 context 的信任。

**具體建議修法：**切案時立即清空所有 project-scoped slices，設 `projectContextVersion`；任何舊請求回來時若 version／project id 不符就丟棄。頁面在新案資料完成前只顯示帶專案名的 skeleton，不可保留舊值。加 e2e：有資料案 → 空案，逐頁 assert 前案唯一標記為 0。

### P0-08｜正式版契約檔上傳的前端 path 與已部署 Storage policy 不一致

**頁面 + 角色 + 資料狀態：**契約與文件；施工／專案管理者；全新案、無契約。

**問題：**依正式前端 `storagePathFor()` 產生 `projects/{project}/contract-packages/{package}/{document}/{version}/{filename}` 上傳到 production bucket，正式版回：

> invalid input syntax for type uuid: "projects"

用 `{projectId}/{packageId}/...` 才能通過目前 deployed policy。repo migration 註解與 source 都預期 `projects/.../contract-packages/...`，代表 production policy／migration 與部署 artifact 不一致。

**為什麼嚴重：**新案第一個關鍵流程「上傳契約 → 抽履約需求 → 後續 AI grounded」被擋，後面所有 AI 不是無資料就是回退。雖根因屬部署／資料層，本報告標註為「後端／部署 bug」，但對使用者是 P0 UX blocker。

**具體建議修法：**對 production 查實際 `pg_policies`，把 folder segment 與 source migration 統一；部署後以真 file input 做 smoke test。上傳失敗 UI 要把「儲存權限設定錯誤」與「檔案格式錯誤」分開，並保留可重試佇列。

### P0-09｜Copilot 對矛盾金流回答「第 5 期無未收款」

**頁面 + 角色 + 資料狀態：**全站 Copilot；監造；完整案、金流已有矛盾資料。

**輸入：**

> 本案第5期累計估驗金額、已收款與未收款各多少？只用專案資料回答並列出出處。

**AI 回覆（12.3 秒）：**

> 第 5 期……實收金額：新臺幣 9,999,999 元。  
> 第 5 期無未收款。……「已請款未收款期」欄位為空陣列，表示目前所有期次均無待收款項。  
> 「累計估驗」欄位數值為 5,908，但該欄位含義……未加詳細說明。

出處連結 `金流-逐期資料`、`金流-已請款未收款期` 都只導到 `#/payments`。

**錯在哪：**專案資料同時是累計估驗 `5,908`、實收 `9,999,999`、UI 未收 `-9,999,999`、狀態仍待請款。AI 應先指出資料不一致，不能把空陣列推論成「所有期次均無待收」。它也沒有回答使用者要求的三個數字。

**具體建議修法：**Copilot 的 facts snapshot 先通過財務 invariant；矛盾時回「資料異常，無法判定」，列出原始欄位。問句中的每個 requested slot 要有 coverage check。出處應深連結到第 5 期並帶欄位 anchor，而不是只到模組首頁。

## 4. P1：明顯卡體驗或流程不完整

### P1-01｜帶入天氣今天與過去日期都只顯示 Edge Function generic error

**頁面 + 角色 + 資料狀態：**施工日誌；施工；已有座標。

**問題：**2026-07-13 約 7.27 秒、2026-06-10 約 3.80 秒，兩者都顯示：

> 天氣未帶入：Edge Function returned a non-2xx status code

沒座標時雖可展開座標設定，但過去日期沒有產品規格所說的友善提示。

**建議：**解析 edge response body 的業務錯誤碼；過去日期顯示「中央氣象署即時資料不支援此日期，請手動填寫／改用歷史觀測」。今天服務失敗顯示來源、重試與手動輸入，不暴露 SDK 原文。

### P1-02｜照片覆核只能取消配對，不能改選正確工項；失敗張的文案與行為互相矛盾

**頁面 + 角色 + 資料狀態：**施工日誌照片 staging；施工；部分成功／部分失敗。

**問題：**caption 可改、單張失敗不會拖垮其他張，這是好設計；但配錯工項只能按 `✕` 取消，沒有 WorkItemPicker。錯誤張顯示「仍可自行填說明後上傳」，實作卻在 `SiteLog.jsx:225` 對 `status=error` 直接跳過；使用者填完也上不去。

**建議：**每張都提供可搜尋工項 picker；error 狀態加「改為手動上傳」把狀態轉 `manual_ready`。上傳後保留未成功的 staging，不要把整個陣列清空。

### P1-03｜Copilot grounding 快照漏掉已存在的契約需求

**頁面 + 角色 + 資料狀態：**Copilot；監造；完整案已有需求「每月 5 日前提送施工月報」。

**輸入：**要求引用原文、頁碼與需求名稱。

**回覆（12.3 秒）：**安全地說「快照未收錄，無法直接回答」，並導到契約與文件／施工月報。沒有亂編，值得肯定；但同專案需求頁與文件 AI 已有該條的頁碼、原文與標題。

**建議：**assistant snapshot 納入核定需求的 title、original_text、page、source document；依權限控制，不要只放 route summary。回覆「無資料」前做一次需求索引查詢。

### P1-04｜建立專案表單預填真實感很強的別案資料，容易直接誤存

**頁面 + 角色 + 資料狀態：**建立專案；首次使用者；無專案。

**問題：**預填「國際原住民族文化創意產業園區新建工程」、契約 `20200710`、桃園市／桃園市政府、2026-01-15～2027-06-30，施工廠商又預設成建立者的機關公司。

**為什麼是 UX 問題：**這些不像 placeholder，而像已存在的正式資料；使用者很可能只改專案名就建立出錯的契約、機關、廠商與工期。

**建議：**正式站所有欄位預設空白，用 example helper text；施工／監造／機關 parties 分開選，不能拿登入者公司填到施工廠商。Demo 才可一鍵套範例。

### P1-05｜空案多數頁只解釋「沒有」，沒有可執行的下一步或 ball-in-court

**頁面 + 角色 + 資料狀態：**施工日誌、估驗、成本、進度、品質、送審、變更、報表；三角色；空案。

**問題：**Dashboard 有「先到契約與文件」的阻斷式 CTA，是好 pattern；但其他頁多半只留說明文字或空表。沒有說清楚「缺 PCCES／缺日誌／缺施工角色／輪到誰」，也沒有回到前置步驟的按鈕。

**建議：**建立共用 `PrerequisiteEmptyState`：缺什麼、誰能補、完成後會解鎖什麼、單一 primary CTA。對唯讀角色顯示「等待施工廠商上傳，已通知／複製邀請連結」，不要導向自己沒有權限的頁面。

### P1-06｜機關在日誌／品質看到可輸入外觀，最後才知道不能存；文案還叫他「監造帳號」

**頁面 + 角色 + 資料狀態：**施工日誌、品質管理；機關；空案與完整案。

**問題：**多個 input 視覺上可編，但沒有儲存動作；日誌底部統一寫「監造帳號唯讀，日誌由施工廠商填報」，機關帳號也看到同一句。

**建議：**唯讀模式直接 render 靜態值或 disabled fieldset，頁首顯示「機關監督檢視」。文案依 `org_type`：機關、監造分開，不要硬編角色名。能填但不能存是最差的 affordance。

### P1-07｜金流可形成負未收、無請款／收款日卻已有實收，狀態仍待請款

**頁面 + 角色 + 資料狀態：**估驗與金流 → 請款收款；機關；完整案。

**問題：**累計應收 `0`／或本案核定基準 `5,908`，實收 `9,999,999`，UI 顯示未收 `-9,999,999`，但狀態仍「待請款」，請款日與收款日皆空。

**標註：**資料完整性／狀態機 bug，已由同日正式版回歸報告與本輪 Copilot 問答交叉看到；對本輪重點的影響是 UI 與 AI 都把不可能狀態當正常事實。

**建議：**未請款不得填實收；實收必須同時有收款日，且不得小於 0 或無限制超過核定應付。負未收顯示為資料異常，不得當 KPI。用單一 RPC 原子寫入並加 DB constraint。

### P1-08｜375px 佐證包整頁寬 485px，表格與頁首一起水平漂移

**頁面 + 角色 + 資料狀態：**估驗佐證包；施工／監造；完整案；375 × 812。

**問題：**DOM `clientWidth=360`、`scrollWidth=485`。頁首三個操作擠成多行；工程資訊欄、AI 說明與七欄表格造成 document-level 橫向捲動。使用者要左右拖整頁才能讀同一列，簽章／照片區也會失去位置感。

**建議：**手機改為卡片式工項：工項名、數量、金額、佐證垂直排列；列印版才使用 A4 表格。若保留 table，只讓 table container 橫捲，頁面本身 `overflow-x:hidden`，首欄 sticky 並加捲動提示。

### P1-09｜超長 RFI 直接完整展開，待辦列表被單筆資料淹沒

**頁面 + 角色 + 資料狀態：**工程疑義；監造；完整案、有 2,600 字 RFI。

**問題：**標題與問題全文在卡片中重複多次、無預覽截斷；一筆測試資料佔數個螢幕，其他待回覆項目與 action 很難找到。手機更明顯。

**建議：**列表只顯示 2–4 行摘要與 metadata；詳情用可展開區／detail route。保留「球在誰手上、期限、工期／費用」固定在卡片頂部。

### P1-10｜AI 長任務只有單一 busy；失敗時有些入口直接留下空白

**頁面 + 角色 + 資料狀態：**送審讀文件、估驗佐證包；監造／施工；完整案。

**問題：**讀兩頁 PDF 約 26.3 秒，只有等待狀態，沒有解析頁數、比對需求、可取消或背景通知。佐證包「重新產生施工說明」曾結束後 textarea 仍空白，沒有錯誤、重試原因或 request id；同一 production endpoint 直接呼叫其實有回覆。

**建議：**AI panel 統一 `idle → uploading → extracting → grounding → generating → success/error`；超過 8 秒顯示目前階段，允許離頁後通知。失敗保留舊結果，顯示可重試錯誤與可複製 request id，不可只留空白。

### P1-11｜正式空案仍有「用範例標單試試」，會把 demo 資料混入正式專案

**頁面 + 角色 + 資料狀態：**標單工項；施工／管理者；空案。

**問題：**主要 CTA 已正確導向專案文件，但旁邊仍有 active「沒有檔案？用範例標單試試」。新手可能把範例 3,262 工項當真資料，再往下建立估驗、日誌與進度。

**建議：**正式站移除；若保留，必須明示「建立沙盒，不可轉正式」，另建 demo project，不得寫入正式案。空案也先隱藏 0 KPI、checkbox、表格 legend 等無用控制。

### P1-12｜機關仍看到重新匯入標單、組佐證包、產生排程等主動 action

**頁面 + 角色 + 資料狀態：**標單、估驗、進度；機關；完整案。

**問題：**機關可見 active `↻ 重新匯入標單`、`組請款佐證包`，進度頁也有產生排程 affordance；有些可按、有些按後沒有可持久權限，角色邊界不一致。尤其重匯標單會清相依資料，雖有確認框仍不應對監督角色出現。

**根因線索：**BOQ 的重匯只判 `dbMode && workItemsSource==='db'`，沒有 `can.edit/admin`。

**建議：**每個 action 用 domain permission 判斷，不以「頁面可讀」推導「動作可做」。機關需要的是檢視、核定、匯出與稽核；破壞性資料維護只給 admin／施工資料管理角色。

### P1-13｜排程日期有 optimistic 外觀但 F5 消失，沒有成功／錯誤回饋

**頁面 + 角色 + 資料狀態：**進度管制；施工；部分／完整案。

**問題：**同日另一組正式版真實操作兩次重現：日期 input 當下保留、無錯誤，F5 後兩欄皆空，DB row 日期仍 null。本輪未重做第三次寫入，只把它列為仍影響本次 UX 結論的已知未解問題。

**建議：**controlled state + `await upsert(...).select()`；server 沒回列就 rollback 並顯示紅字。不要把 DOM input 的暫存值當保存成功。

### P1-14｜手機 Copilot 打開後佔滿工作畫面，且長對話缺少 context／回到頁面的捷徑

**頁面 + 角色 + 資料狀態：**全站 Copilot；三角色；375px、已有多輪對話。

**問題：**面板變成近全螢幕且內部再捲動；雖有頁首 `×` 和右下收合鈕，但使用者無法一邊看來源頁一邊核對答案。出處只到模組首頁，關閉後也不會定位到第 5 期或需求。

**建議：**手機改 bottom sheet，預設 70–80vh；點出處先收合並 deep-link 到實體資料。保留目前頁／專案 context badge，讓使用者知道回答針對哪案。

## 5. P2：打磨與一致性

### P2-01｜活動紀錄要使用者輸入 UUID

**頁面 + 角色 + 資料狀態：**活動紀錄；三角色；空／部分／完整。

**問題：**篩選 placeholder 是「執行者 UUID（或 system）」。即使有 datalist，這仍是開發者語言，不是工程管理者語言。

**建議：**改為可搜尋的人名／公司／角色 picker；`system` 顯示「系統」。UUID 只放 debug detail。

### P2-02｜Copilot 把 Markdown 星號原樣顯示

**頁面 + 角色 + 資料狀態：**Copilot；監造；完整案。

**問題：**回答顯示 `**第 5 期估驗資訊：**`、`**無法直接回答**`，沒有 render 成粗體。這會讓真 AI 看起來像 debug output。

**建議：**要嘛安全 render Markdown allowlist，要嘛 prompt 明確禁止 Markdown 並在輸出層清理。

### P2-03｜送審 caution 顯示 literal `\\n`

**頁面 + 角色 + 資料狀態：**送審審查助手；監造；完整案。

**問題：**三點 caution 中間直接看到 `\\n` 字元。

**建議：**schema 使用 string array；UI 逐項 `<li>` render。若後端維持 string，先正規化 escaped newline。

### P2-04｜手機頂列專案名截斷但沒有辨識補救，部分命中框仍小於 44px

**頁面 + 角色 + 資料狀態：**全站；三角色；375px。

**問題：**專案名只剩 `PMIS-R4-甲-正式…`，使用者在多個相似案難辨識；menu、登出、表格小型刪除／修改仍偏小。同日 390px 回歸量測最小可到約 `11 × 20`。

**建議：**頂列顯示短代碼或可點開完整專案 chip；所有重要與破壞性 action 的可點區至少 44 × 44，文字不必放大。

### P2-05｜月報允許填檢討／下月計畫卻明示「不會儲存」

**頁面 + 角色 + 資料狀態：**報表中心 → 施工月報；施工／監造；部分／完整。

**問題：**欄位看起來像正式編輯，底部才說「可臨時填寫後列印（不會儲存）」。離頁、重整或換裝置就消失。

**建議：**要嘛做草稿自動儲存與版本，要嘛改成「列印前暫填」dialog，明顯標示 ephemeral，避免和正式報表混淆。

### P2-06｜固定金額罰款常見格式無法試算，但 UI 沒說為什麼

**頁面 + 角色 + 資料狀態：**契約與文件；施工／監造；有逾期義務。

**問題：**`每日按契約總價千分之一` 能正確算出 12 日約 `NT$8,656,369`；抽不出罰率時也會安全地不顯示，這是好行為。但 `新臺幣 10,000 元／日` 這種含逗號與空白的常見固定金額沒有試算，也沒有顯示「未辨識罰率」。

**建議：**parser 支援中文數字、逗號、空白、`元/日`／`元每日`；無法解析時顯示「偵測到罰則，但金額格式需人工確認」，不要默默消失。

### P2-07｜需求空頁先顯示六個 filter 與審查面板，資訊架構倒置

**頁面 + 角色 + 資料狀態：**契約需求／文件；監造／機關；空案。

**問題：**0 筆需求時仍先看到六個 filter、空審查區與大量 chrome，真正的下一步不突出。

**建議：**空案只顯示上傳契約、處理狀態與角色責任；有第一筆資料後再展開 filter 和批次審查工具。

| 變更設計 | 🟡 | 🟢 | 🟢 | 角色化核准流程可理解；空案缺前置與「輪到誰」。 |
| 驗收結算 | 🟢 | 🟢 | 🟢 | 前置條件、階段與角色 gate 最完整，值得複用。 |
| 報表中心 | 🟡 | 🟡 | 🟡 | 空報表缺前置；月報暫填不儲存；跨頁矛盾資料照單彙整。 |
| 專案成員 | 🟢 | 🟢 | 🟢 | 邀請／角色／正式模式相對清楚；Demo 也明示不會真邀請。 |
| 風險稽核（機關） | 🟢 | 🟢 | 🔴 | 空案會說明依賴，`非違規認定` 很好；AI 處置建議越權。 |

## 8. 第一次使用旅程

目前第一次使用最像這樣：

1. 建立專案時先被一組像正式資料的預填值誤導。
2. 建完專案，Dashboard 正確說先去契約與文件。
3. 建立者若是機關身分，契約頁又說需要施工／管理編輯權，但沒有清楚的「邀請誰、球交給誰」。
4. 正式上傳 path 被 production policy 擋，AI grounding 鏈中斷。
5. 使用者如果轉去標單頁，會看到「用範例標單試試」，很容易把 demo 資料灌進正式案。
6. 其他頁多為「0 張／0 筆／尚無」與 disabled-looking 空表，沒有統一解鎖路徑。

建議把 onboarding 做成專案級 checklist，而不是每頁各自猜：

1. 確認三方 parties 與成員。
2. 施工／管理者上傳契約；AI 擷取後由監造／機關覆核需求。
3. 上傳 PCCES；顯示工項數與總價核對。
4. 設工地座標與開工日。
5. 建第一筆日誌與照片。
6. 達成前五項後才開放估驗、報表與稽核 AI。

每一步顯示責任角色、目前負責人與「通知／邀請」動作。對無權限者不要顯示死按鈕，改顯示球在誰手上。

## 9. 做得好、值得推廣的 pattern

1. **RFI 的設計拒答：**對切筋與補強長度明確說資料不足、轉設計／結構技師，不硬編答案；還標工期與費用影響。
2. **文件閱讀的引用結構：**逐項 `符合／部分符合／未涵蓋／需人工確認`，能引用頁面與原文；刻意缺漏內容沒有假裝符合。應保留此 schema，只修適用範圍。
3. **罰款試算的 safe fail：**`千分之一` 能正確處理；抽不到罰率時不亂算，並顯示「概算供參」。
4. **照片 staging 的逐張隔離：**三 worker 逐張完成，單張失敗不拖垮整批；caption 先覆核再落庫。只差工項 picker 與 error/manual 狀態。
5. **驗收結算的 prerequisite + role gate：**清楚說明階段、期限、誰能記錄下一步，沒有一堆此刻不能用的按鈕。
6. **工安唯讀與風險稽核 disclaimer：**「值得複查的異常提示，非違規認定」位置清楚。下一步是限制正文不可越過 disclaimer。
7. **Dashboard 空案 CTA：**直接告訴使用者先到契約與文件，是全站空狀態應統一採用的結構。
8. **破壞性操作確認：**重匯標單等動作已有明確影響範圍與二次確認；應再加正確角色 gate。

## 10. 建議修復順序與驗收條件

### 第一批：先停止錯誤正式輸出

- 暫時關閉估驗佐證包列印／AI 自動說明，直到 P0-01、P0-02 修完。
- 工安 AI 移除無來源數值；風險稽核禁止技術／契約處置詞。
- 送審需求加適用文件類型，scope 不符不得影響建議判定。

驗收條件：`2 M3 × 2,954` 在主頁、佐證包、AI payload、列印皆為 `2 / 5,908`；負向照片預設不入包；AI 不得出現未在 source 中的尺寸與剔除／補強。

### 第二批：修 project context 與新案解鎖

- 切案清 slice、丟棄舊 request response。
- 修 production contract-documents Storage policy。
- 建 onboarding checklist 與共用 prerequisite empty state。

驗收條件：完整案 → 空案後，不刷新也找不到前案唯一標記；真 PDF 經 file input 上傳、處理、抽需求、F5 全保留；三角色各自只看到能做的下一步。

### 第三批：AI grounded contract

- 建統一 fact schema、source id、scope、confidence、human decision fields。
- AI 前做 deterministic invariant，AI 後做 forbidden-claim／requested-slot coverage validator。
- 出處改成可定位 entity 的 deep link。

驗收條件：十項對抗測試固定進 CI／staging；每一項都必須能判斷真 AI、看見階段、失敗可重試、輸出有來源、人工覆核後才成為正式紀錄。

### 第四批：手機與一致性

- 佐證包 mobile card layout；表格只在容器橫捲、sticky first column。
- 44 × 44 命中框；專案 chip 顯示可辨識短碼。
- AI／空狀態／錯誤 panel 統一 component 與繁中詞彙。

驗收條件：375px 全站 document-level `scrollWidth <= clientWidth`（明確指定的 table scroller 除外）；主要／危險 action 命中框至少 44 × 44；唯讀角色沒有可輸入假象。

## 11. 仍需補測

1. 使用可控 `setInputFiles` 的 Chromium／Playwright，補契約 PDF、DOCX、PCCES、壞檔、大檔、同檔重送與上傳中斷。
2. CWA 正常回應、過去日期、無座標、timeout、500、離線與重試。
3. 一批 10–20 張照片混合成功／失敗／非工地，在 375px 完成改 caption、改工項、排除、重試與上傳。
4. iOS Safari、Android Chrome、相機／相簿權限、軟鍵盤與橫直轉向；本輪 375px 是桌面瀏覽器 viewport，不等同實機。
5. 真正多頁列印：佐證包、施工月報、監造報表、長表跨頁表頭與簽章區。
6. 修復後重跑本報告所有 P0 對抗 prompt，不能只看單元測試或 Demo 確定性回退。
