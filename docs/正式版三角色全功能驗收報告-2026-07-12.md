# PMIS 正式版三角色全功能驗收報告

> 本報告取代 `全角色全功能驗收報告-2026-07-12.md` 的 Demo 分析。此次只保留三種角色：施工廠商、監造、機關；所有結論以正式版、正式 Supabase 資料與真實 PCCES 標單為準。

## 1. 驗收摘要

- 驗收日期：2026-07-12
- 正式版網址：`https://ryanxxhuang.github.io/PMIS/`
- 獨立驗收專案：`QA-正式版全功能驗收-20260712-B`
- 工程代碼：`QA-20260712-002`
- 三角色帳號：施工廠商、監造、機關各一組正式帳號
- 真實標單：PCCES XML，共 3,262 筆節點、2,801 筆可計價末端工項
- 發包工程費：NT$721,364,067
- 驗收方式：實際新增、修改、送審、退回、核定、判定、查詢、重新登入、跨角色接手及正式資料回寫；不是只查看頁面。

### 整體結論

正式版已能支撐「施工日誌 → 品質／取樣 → 估驗 → 月報 → 儀表板」的一部分主幹，也能保存真實大標單、跨角色待辦及活動紀錄。但目前不宜直接作為正式工程的完整簽核系統，原因不是單純 UI 不好看，而是有三類會影響履約證據與金流正確性的核心問題：

1. 監造的送審核准、RFI 回覆、一般查驗不合格等按鈕在正式版無反應。
2. 施工廠商建立者取得過大的完整權限；機關也能由網址直接進入成本頁，畫面權限與資料庫權限互相矛盾。
3. 未核定估驗仍可填實收金額，估驗退回又不要求退回原因，會讓金流與審查證據失真。

建議先完成 P0 修復，再做 UI 美化或增加新 AI 功能。

## 2. 結果統計與判定定義

- **通過**：實際操作、正式資料回寫、重新載入後仍存在。
- **部分通過**：主流程可用，但有重要欄位、權限、狀態或回饋問題。
- **失敗**：實際按下後無反應、狀態不變，或產生錯誤資料。
- **受限／未完整驗證**：受瀏覽器測試工具檔案選擇限制，不能據此判定為系統 Bug。

本次共涵蓋 37 個功能區／跨模組能力：10 項通過、17 項部分通過、7 項失敗、3 項受限或受前置流程限制。

## 4. P0：上線前必修

### P0-1 監造核心協作按鈕無法使用

正式監造帳號實測：

- 一般品質查驗按「不合格」：按鈕只有焦點狀態，查驗仍待查驗，也沒有自動開缺失。
- 送審文件按「受理審核」或「核准」：狀態仍為待監造審定。
- RFI 按「回覆」：沒有回覆欄位、對話框或狀態變化。

影響：品質、送審、設計釋疑三條核心監造流程中斷，系統不能作為正式履約往來證據。

建議：把狀態轉移集中到同一個 workflow service；每一動作必須有 loading、成功 toast、失敗錯誤、活動紀錄與重新讀取驗證。建立端到端測試：廠商提出 → 監造退回／不合格 → 廠商補正 → 監造核准／結案。

### P0-2 角色權限與畫面宣告互相矛盾

「專案成員」頁明確寫主辦機關唯讀，但實際發現：

- 施工廠商建立者可看到並執行估驗退回／核定、送審核准／核備／退回、RFI 回覆、監造工安查驗、建立 ITP 停留點與變更狀態核准。
- 機關可直接把變更設計狀態改成核准，合理；但同時仍可編輯數量、單價、刪除明細與新增明細。
- 機關導覽雖隱藏成本管理，直接輸入 `#/cost` 仍能進入完整頁面與新增表單。

影響：任何建立專案的廠商都可能替監造／機關做出正式判定；網址隱藏不等於權限控制。

建議：仍維持三角色，不需要先細分職稱；但後端與前端必須使用同一份能力矩陣。專案建立者只保留成員管理與專案設定，不得自動取得監造／機關的法定簽核能力。所有路由需在進頁前驗證能力，所有 mutation 再由 RLS／RPC 二次驗證。

### P0-3 未核定估驗可形成已收款

第 1 期估驗尚在監造審核時，施工廠商可填實收 NT$28,063；儀表板與月報立即顯示累計已收。之後監造把該期退回草稿，已收款仍存在，活動紀錄也寫入「請款／撥款資料更新」。

影響：估驗狀態、應收、已收、實際進度與會計資料互相矛盾。

建議：只有核定估驗能產生請款；只有請款完成後才能登錄收款；退回／撤銷核定時必須阻擋或反轉後續金流。日期與金額應同一交易保存，不能金額成功、日期失敗。

## 5. P1：資料正確性、效能與流程完整性

### P1-1 AI 月報產生錯誤事實

AI 草稿寫出：

- 「實際進度率 0.41%」；實際 NT$29,540 ÷ NT$721,364,067 約為 0.0041%，畫面也顯示 0.0%。
- 「辦理 2 場監督檢驗，合格率 100%」；當時一筆合格、一筆仍待查驗，並非 2 筆都合格。
- 把本期估驗金額直接稱為「本月完成工作量」。

建議：AI 不自行算數；所有比率、件數與狀態先由程式產生結構化 facts，AI 只能把已驗證 facts 改寫成文字。生成後跑 deterministic validator，若文字數字不在 facts 中就阻擋並標紅。

### P1-2 風險稽核給出錯誤安全感

- 沒有任何契約／履約需求資料，卻顯示「契約履行：無逾期義務／通過」。正確狀態應是「資料不足，無法判定」。
- 只有一期估驗也顯示「各期金額變化平穩」。至少需要兩至三期才能做趨勢判定。
- 缺失沒有明確期限，卻宣稱都在改善期限內。

建議：所有風險規則增加 minimum evidence；資料不足顯示灰色「未評估」，不可算入通過數。

### P1-3 真實 PCCES 標單造成估驗卡頓

2,801 個末端工項下：

- 建立第 1 期估驗約需十秒級才出現。
- 搜尋單一工項也出現長時間卡頓。
- 頁面截圖甚至曾因渲染繁重逾時。

建議：估驗明細不要一次建立／渲染所有列。採虛擬列表、伺服器搜尋、只展開已估驗或搜尋命中的節點；父層金額用 materialized summary 或後端聚合。

### P1-4 缺少必要的審查證據

- 估驗退回不要求原因。
- 查驗合格不要求查驗紀錄、照片、量測或監造意見。
- 缺失開始改善與提送複查不要求改善說明、前後照片或完成日。
- 變更設計只用下拉選單改狀態，沒有簽核意見、依據、公文號或核定日。

建議：所有跨角色狀態轉移都必須開啟一致的「動作面板」：意見、必要附件、日期、執行者、前後狀態與確認；並寫入不可變活動紀錄。

### P1-5 自動帶入不完整

- 施工日誌頁說明明確宣稱估驗可按「從施工日誌帶入」，估驗頁實際沒有此按鈕。
- 取樣試驗可以從日誌建立，但沒有由 `結構用混凝土，280kgf/cm²` 帶入 fc′。
- 逐工項排程已填起迄，仍顯示未排定。

建議：把日誌工項、查驗、取樣、估驗、排程全部以同一 `work_item_id` 串接；不要再靠文字摘要猜測。估驗預填後由廠商確認差異，監造只看異常項。

### P1-6 日期輸入失敗沒有回饋

請款日、收款日與尚未報竣的驗收日期輸入後被清空；畫面沒有說明是前置條件、格式錯誤、權限不足或儲存失敗。

建議：不可默默重設欄位。若因前置階段鎖定，直接 disabled 並顯示「需先完成報竣／核定」；若 API 失敗，保留使用者輸入並顯示欄位級錯誤。

### P1-7 跨報表定義不一致

施工月報顯示本月雨天 0 天；監造報表對同一筆日誌顯示雨天 1 天。日誌上午晴、下午陣雨，兩份報表使用不同判定邏輯。

建議：定義雨天工日、半日雨天、停工日等共用計算規則，報表只取同一個後端指標。

## 6. 大量手動填寫與實務現場優化

### 施工日誌

目前一天要重填人員、機具、材料、技術士、安衛、取樣與通知事項。實務上工地主任／品管／職安會在手機、日照、手套、網路不穩的情境下操作，不可能每天逐欄重建。

建議優先順序：

1. 「複製昨日」並只提示差異；人員、機具、常用材料可設班組範本。
2. 由出勤名冊、機具清單、材料進料單自動帶入，使用者只確認數量。
3. 天氣由工地座標自動帶入上午／下午，允許人工修正並保留來源。
4. 手機語音輸入與照片 OCR 先生成草稿，再由人確認；不要直接寫正式紀錄。
5. 公定格式欄位不要藏在展開區後再逐列新增，應提供「今天有變動的項目」簡化模式。

### 估驗與請款

- 預設從已保存日誌累計、合格查驗與核定變更計算建議數量。
- 只顯示「本期有變動、超過契約量、缺查驗佐證、與日誌不一致」的工項。
- 監造審核以差異為中心，而不是重新閱讀 2,801 筆。
- 估驗核定後自動產生請款草稿；收款由銀行／會計匯入或至少以核定應領為預設。

### 品質、ITP、工安

- ITP 申請查驗時自動建立品質查驗，不要讓使用者在兩頁追同一件事。
- 日誌填「混凝土 10 M3」時自動建立試體、帶 fc′、取樣數量與 7／28 天期限。
- 缺失改善必須支援現場拍「改善前／改善後」，AI 可比對是否同位置，但最後仍由監造判定。
- 工安自主檢查應先選檢查表或合格／不合格結果，不應新增後一律待改善。

### 文件、送審與 RFI

- 不要只存「附件說明」。應支援真正版本檔、雲端連結、版次、雜湊、送審公文號及下載權限。
- RFI、送審、變更可共用圖面標註與附件元件，回覆時可引用圖號／規範條文。
- AI 從契約／規範擷取的內容必須附頁碼與原文片段，人工核定後才成為提醒與檢查表來源。

## 7. UI／UX 與邏輯問題

1. 建立者看到所有角色按鈕，使用者無法理解自己目前是在「施工廠商」還是「系統管理者」身分操作。
2. 很多動作只有按鈕取得焦點，沒有 loading、成功、失敗或狀態改變；使用者會反覆點擊。
3. 估驗說明寫有日誌帶入功能但按鈕不存在，屬功能承諾與 UI 不一致。
4. 變更以可任意切換的下拉選單表示正式狀態，不符合簽核流程心智模型。
5. 工安自主檢查沒有結果欄，新增後卻直接顯示待改善。
6. 逐工項排程有日期卻仍未排定，狀態與欄位互相矛盾。
7. 月報的 AI 草稿不保存，使用者若離頁會遺失編修；至少要有草稿保存與版本。
8. 監造、施工月報與 Dashboard 對同一指標採不同定義，使用者無法信任數字。
9. 大表格缺少虛擬捲動、固定欄、只看異常、批次填入及鍵盤操作。
10. 直接路由可進隱藏功能，顯示「導覽隱藏」被誤當成權限控制。

## 8. AI／自動化應做與不應做

### 應做

- 天氣、日誌、出勤、材料與機具預填。
- 從 work item、規範與品質計畫產生 ITP、取樣、檢查表草稿。
- 估驗差異偵測、超量、缺佐證、重複計價、異常單價提示。
- 文件分類、版本差異、條文頁碼引註、期限建議。
- 月報先由程式算好 facts，再由 AI 撰寫。
- 對每個待辦提供「為什麼輪到你、下一步、期限、依據」而不是只顯示件數。

### 不應做

- AI 自行計算金額、百分比、件數或合格率。
- 在沒有契約資料時宣稱沒有逾期。
- 沒有足夠期數時宣稱趨勢平穩。
- AI 直接核定、送審、判定合格或改正式紀錄。
- 產生沒有出處、頁碼或資料時間點的風險結論。

## 9. 系統瘦身與功能整併

### 建議整併

1. **估驗計價 + 請款收款**：整成「估驗與金流」，用狀態分頁，不需要兩套各自可編輯的資料。
2. **進度 S 曲線 + 逐工項排程**：整成「進度管制」，上層 S 曲線、下層關鍵工項與要徑。
3. **品質查驗 + ITP + 自主檢查 + 取樣試驗**：共用一個「品質管理」工作台，以查驗類型分頁；ITP 是查驗來源，不應成為孤立模組。
4. **品質觀察／缺失 + 工安觀察／缺失**：共用一個事件／缺失引擎，再用品質或工安分類，避免兩套改善狀態機。
5. **送審文件 + RFI + 變更設計**：共用「技術協作」工作台與附件、版次、回覆、球在誰手上機制。
6. **專案文件 + 履約需求 + 風險稽核**：整成「契約與文件智慧中心」；先有資料，再做需求與風險。
7. **施工月報 + 監造報表 + 驗收文件**：整成「報表中心」，共用指標、版本與簽章。
8. **Dashboard + 提醒中心 + AI 助理主動分析**：Dashboard 顯示待辦與風險，AI 助理保留問答；不要三處重複顯示同一件事。

### 建議隱藏或延後

- 沒有契約文件時隱藏「風險稽核通過數」，改顯示資料準備度。
- 尚未報竣時，驗收結算只顯示前置檢核與報竣，不用一次展示所有後續空表單。
- 施工廠商不要看到監造報表、監造工安事件、估驗核定、送審核准、RFI 回覆等控制。
- 機關不要看到成本管理與可編輯的施工細節；只看契約金額、變更、估驗、支付、進度與風險。
- 在 AI 正確性驗證完成前，不再增加更多零散的「AI 草擬」按鈕。

## 10. 額外補充的評估面向

### 資料完整性

- 所有狀態轉移需 server-side transaction。
- 金流與估驗狀態建立外鍵／檢核限制。
- 報表指標集中計算並帶 `as_of` 時間。

### 稽核與法律證據

- 活動紀錄已有良好基礎，但執行者應顯示真實姓名、三角色身分、組織與當時權限快照，不可硬編碼成專案經理。
- 退回、核定、不合格、結案、撤銷都要保存理由與附件雜湊。
- 已完成紀錄更正要新增版本，不覆寫原證據。

### 行動裝置與工地環境

- 本次以桌面瀏覽器為主，手機相機、弱網、離線暫存、手套操作與陽光可讀性未完成實機驗證。
- 建議下一輪用 iPhone／Android 實機，在 4G 弱網下驗證日誌、查驗、缺失照片與簽名。

### 可用性與無障礙

- 重複的無名稱日期、數字欄位讓自動化與螢幕閱讀器難以分辨。
- 狀態不可只靠顏色；需文字、圖示與 aria label。
- 大表格應支援固定欄、鍵盤移動、批次貼上與錯誤定位。

### 測試工具限制，不能誤判為產品 Bug

- 內建瀏覽器明確不支援檔案上傳；Chrome 連線也拒絕 `setFiles`，因此 PCCES XML 以專案既有解析器解析，再透過正式 Supabase API／RLS 寫入驗收專案。
- 契約 PDF、規範、圖說、現場照片與 AI 圖像辨識的 UI 檔案選擇流程未完成端到端驗證。
- 列印／CSV／整案匯出的入口已確認，但未逐一核對下載檔案內容與版面。

## 11. 建議修復順序

### 第 1 週：止血

1. 修復監造「不合格、送審受理／核准、RFI 回覆」。
2. 移除建立者跨角色簽核特權；補路由與 mutation 能力檢查。
3. 禁止未核定估驗形成請款／收款，處理既有矛盾資料。
4. 所有失敗動作顯示錯誤，不可無聲失敗。

### 第 2–3 週：流程證據

1. 統一退回／核定／判定動作面板與必填理由。
2. 補改善說明、前後照片、查驗量測與附件版本。
3. 修正逐工項排程狀態及日期欄位回饋。
4. 統一雨天、進度、查驗件數等報表指標。

### 第 4–6 週：減少手填與效能

1. 估驗從日誌自動帶入，改成只審差異。
2. PCCES 大表格虛擬化與伺服器搜尋。
3. 日誌複製昨日、班組／機具／材料範本、天氣自動帶入。
4. 取樣自動帶 fc′ 與試體數量。

### 第 7 週後：瘦身與可信 AI

1. 依第 9 節整併導覽與工作台。
2. 建立結構化 facts + validator 後再恢復 AI 月報／稽核為正式功能。
3. 做手機弱網實機與完整文件上傳回歸測試。

## 12. 最終驗收判定

- 施工填報與資料彙整：**可試辦，但需修正效率與資料串接。**
- 監造正式審核：**不通過；送審、RFI、一般不合格判定存在核心失效。**
- 機關監督與變更核定：**部分通過；核定可用，但只讀與路由權限不可信。**
- 財務／履約證據：**不通過；未核定估驗可形成已收款，退回又無理由。**
- AI：**僅可作草稿參考，不可直接用於正式月報或風險結論。**

正式上線門檻應是：三個 P0 全部修復、以同一三角色帳號跑完「提出 → 退回 → 補正 → 核准／結案」回歸測試，並以真實 2,801 工項標單維持可接受效能。
