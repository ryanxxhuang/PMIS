## 六、依嚴重度排序的問題

### P0-01｜正式模式 stale 分頁可刪除已受理送審

- 分類：**產品 bug／法律證據完整性**
- 路由：`#/submittals` + Supabase `submittals`
- 重現率：1/1
- 步驟：
  1. A 建 SUB-004 並讓頁面停在「已提送」，畫面可見刪除。
  2. B 以自己的角色 session REST 將同筆改成「審核中」，審查意見「B 已受理，A 分頁仍舊」。
  3. A 不重整，從 stale 畫面按刪除並在 app modal 確認。
  4. 檢查錯誤 banner、F5、REST、audit event。
- 預期：DB 依最新狀態拒絕刪除；A 應收到衝突提示並重載。
- 實際：畫面立即消失、無錯誤；F5 後仍不存在。
- 證據：`audit_events` 有 `submittal.deleted`，`before_data.status="審核中"`、`actor_party_type="contractor"`、`actor_is_project_admin=true`、server timestamp `2026-07-13T06:03:55.123901Z`。
- 根因假設（推測）：`deleteSubmittal` 先 optimistic 移除且忽略 Supabase delete 回傳；DB formal-mode guard 只限制審定狀態 UPDATE，未限制 DELETE，RLS 的 `can_write` 仍讓建立者刪除任何狀態。

### P0-02｜混凝土 28 天不合格判定 F5 回退且未開缺失

- 分類：**產品 bug／品質證據鏈**
- 路由：`#/quality`，`test_samples`、`defects`
- 重現率：1/1
- 步驟：建立 fc′ 280 試體，輸入 28 天值 `200, 200, 200`。
- 預期：平均 200 < 280，且單顆 200 < 0.85×280=238，狀態應「不合格」、自動開缺失並持久。
- 實際：畫面立即顯示不合格；F5 後回「待試驗」。DB 讀回 `d28_values=[200,200,200]`、`status="待試驗"`，缺失表沒有該試體缺失。
- 根因假設（推測）：前端先本地 `judgeConcrete`，再 await 開缺失及 PATCH 試體，兩個非交易式寫入可能發生 stale state／覆寫；DB 沒有由 d28 值強制推導狀態的 trigger/RPC，因此半套資料可永久存在。

## 十二、瘦身與修復優先順序

1. **先封 P0 stale DELETE**：formal mode 下送審一經提送即禁止實體刪除；改為撤回／作廢狀態，必填原因；DELETE trigger 應依 DB 最新狀態拒絕。所有 optimistic delete 必須檢查 `.select()` 空列與 error。
2. **試體判定下沉 DB/RPC**：一次交易更新 d28、判定 status、建立 defect、寫 audit；以 unique source 防重複缺失。
3. **統一 DB-first mutation helper**：排程、金流清空、刪除等不得先顯示成功；失敗必有紅色 banner。
4. **期限 anchor 測試表**：initial 合格→final；initial 不合格→fix→reinspect 合格→final，分別單元測試。
5. **金流 domain validation**：日期不得超過合理未來窗；paid_amount≥0 且超應領需阻擋或二次確認；正規化 `-0`。
6. **報表採最小證據原則**：沒有 due_date 就寫「未設定期限／未評估」，不要自動宣稱期限內。
7. **自主檢查改成修訂版次**：不可覆寫舊證據；提供 Rev.N 與差異、重新判定、缺失關聯。
8. **無障礙補名**：所有日期／數字欄以工項＋欄位生成 aria-label。
