## 6. 問題清單（依嚴重度）

### P0-01 送審「修正再送」只在本地成功，重整消失且流程死路

- 分類：產品 bug
- 路由：`#/submittals`
- 穩定重現：是，1/1；B 重登與 DB 均確認
- 步驟：
  1. A 提送 SUB-001。
  2. B 受理後退回補正。
  3. A 按「修正再送」。
  4. 當下 DOM 顯示 `Rev.1`、`待監造審定`。
  5. B 重登或 A 重整；DB 仍是 `revision=0,status=退回補正`。
- 預期：`revision=1,status=已提送` 持久化，B 行動中心收到待審。
- 實際：畫面假成功；對方仍看不到，正常角色鏈卡死。
- 證據：A 當下 DOM `Rev.1 / 待監造審定`；B 新 session DOM `待廠商補正`。後續只能用 B session REST 強制寫回 `已提送, revision=1` 才能繼續核准測試。
- 根因假設（推測，高信心）：`submittals_guard` 把「old.status 為退回補正」也視為監造專屬轉移，導致 A 的再送被拒；前端 `resubmitSubmittal` 先 optimistic 更新，DB 錯誤後不 rollback，頁面呼叫端也未 await／顯示 error。

### P0-02 C 可假修改已核准變更明細，契約額當場變錯，重整才復原

- 分類：產品 bug
- 路由：`#/change-orders`
- 穩定重現：是，1/1
- 步驟：
  1. A 建 R2-CO-001，追加 1 × 100,000。
  2. C 核准，正確變更後契約額為 `721,464,067`。
  3. C 仍看到未 disabled 的數量、單價、刪除按鈕；把數量 1 改為 2。
  4. 畫面立即顯示 `+200,000` 與 `721,564,067`，沒有紅色錯誤橫幅。
  5. 重整後回復 `1 / +100,000 / 721,464,067`。
- 預期：已核准明細完全唯讀；C 也只能改核准狀態，不可改內容。
- 實際：UI 顯示假的契約額，可污染使用者當下判斷、列印或抄錄。
- 證據：修改前後 DOM 金額與重整後 DOM；DB 最終仍是 `qty_delta=1, amount_delta=100000`。
- 根因假設（推測，高信心）：ChangeOrders 未依 `co.status==='核准'` 與角色關閉明細 editor；`updateChangeOrderItem` 先更新 React state，再送 DB，忽略／未上拋 RLS 0 rows 或 trigger error，也沒有 rollback。

### P1-01 退回原因必填不一致，送審與估驗可無有效原因退回

- 分類：流程缺陷
- 路由：`#/submittals`、`#/valuation`
- 穩定重現：是
- 送審：退回 modal 標「必填」，但清空後按鈕仍 enabled，可完成退回；畫面沿用先前「受理」的一般意見，不是本次退回原因。
- 估驗：B 按「退回」直接回草稿，沒有 modal、原因欄或稽核說明。
- 預期：每次退回都強制新原因，並與該狀態轉移一起稽核。
- 實際：法定／契約證據鏈欠缺「為何退回」。

## 9. 該自動化清單

### 已做對且應保留

- 日誌工項數量 → AI 估驗草擬 → 累計金額／保留款／應領自動計算。
- 查驗不合格自動開缺失。
- 品質、工安共用缺失狀態機與撤銷稽核。
- 核准變更自動更新契約額與估驗分母。
- 驗收法定期限自前一階段自動起算。
- 風險稽核在只有 1 期估驗、無契約義務時誠實顯示「未評估」，不灌成通過。
- Dashboard 行動中心的 ball-in-court 在 A/B 切換時正確（B 待審 4；退回後 A 待處理 5）。

### 還要手動重打

- 日誌沒有「複製昨日」、班組／機具／材料範本、天氣服務帶入。
- 公定格式的人力／機具／材料收在折疊區，沒有跨日範本；現場每天重打。
- 取樣雖有「從施工日誌帶入」，但需要日誌材料欄先以一致字詞輸入混凝土，未直接從工項規範／BOQ 自動帶 fc′ 與齡期。
- 查驗與 ITP 分頁各自追蹤，缺少同一工項／停留點的單一事件視圖。
- 送審補正沒有可修改的補正表單或新版附件說明。
- 施工月報與監造報表共用 facts 不完整，雨天欄未共用。

### 過度自動化／危險預設

- 正常自主檢查預設「待改善」。
- AI 月報數字 guard 對 legitimate derived fact `0.1%` 誤判。
- 被伺服器拒絕的已核准變更編輯仍在前端即時計算並顯示新契約額。

## 10. 系統瘦身／整併建議

1. 所有 mutation 統一走「DB first → 以 returned row 更新 UI」；禁止每個 slice 自己 optimistic 更新。共用 `mutationOutcome` 並強制處理 200+0 rows。
2. 把送審、估驗、缺失、驗收的狀態機定義集中成一份前後端共用 transition table，避免 UI 可走、trigger 卻擋的分裂。
3. 品質管理內把查驗、ITP、自主檢查、取樣整成「工項品質事件」工作台，保留深連結但不要讓使用者在四頁重複追同一件事。
4. 施工／監造月報應共用一份 facts DTO；雨天、施工日、缺失、估驗只算一次。
5. 契約文件只保留一個 uploader；`accept`、拖放、文案與分類器支援格式由同一常數產生。
6. 正式模式下先由角色與 record status 生成 read-only presenter，再渲染 controls；不要先顯示 editor 再依賴 API 擋。
7. 驗收頁只展開「現在可辦」的單一階段；歷史階段顯示修改（附原因），未來階段只顯示預計期限。

## 11. 補充面向

### 安全與多專案隔離

- 三個新帳號只讀到加入的新專案；B/C 未加入前沒有專案資料。
- REST 越權未穿透，正式模式單向性有效。
- 建議再補 service-role／JWT 過期／跨專案猜 UUID 的專門滲透測試；本輪未使用 service role。

### 稽核與法律證據

- DB 最終有 38 筆 `audit_events`、6 筆 `defect_audits`。
- 缺失撤銷結案要求原因，證據鏈良好。
- 送審空白退回、估驗無原因退回與驗收可先填未來階段，會削弱法律證據可信度。
- 建議稽核事件記 `transition_from/to`、reason、actor org snapshot、server timestamp、request id；重要簽核不可只存目前狀態。

### 行動現場與無障礙

- 390px viewport：quality、safety 沒有 document-level overflow；site-log main 內部寬 518px，但 document 仍 375px，為內部橫向表格滾動，未把整頁撐破。
- 弱網節流未驗證（工具限制）。
- 多個表格 input 缺 accessible name；日期／金額不能只靠欄位位置辨認。
- 狀態大多有文字，不只靠顏色；此點通過。
