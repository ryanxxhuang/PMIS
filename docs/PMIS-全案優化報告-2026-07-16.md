## 一、Bug(正確性)

### 🔴 B-01 品牌字型在正式站根本沒有載入
`src/index.css` 把 Google Fonts 的 `@import` 放在 `@import "tailwindcss"` 之後;Tailwind v4 展開後這行 `@import` 落在一般規則後面,**Lightning CSS 在 build 時直接把它丟棄**(build log 有 warning;已實測 `dist/assets/index-*.css` 內完全沒有 `@import`)。結果:正式站的 Noto Sans TC 與 IBM Plex Mono(`--font-data` 數據體)全部 fallback 系統字型,品牌字型體系(pmis-brand-design 的核心之一)形同未上線。dev 模式看得到、prod 看不到,所以一直沒被發現。
**修法**:把字型改為 `index.html` 的 `<link rel="preconnect">` + `<link rel="stylesheet">`(順帶解掉 render-blocking),或自行 self-host woff2。

### 🔴 B-02 核准變更設計後,全站金額/進度數字分裂
`applyApprovedChangeOrders` 只套用在 **Valuation / ValuationPrint / ValuationPackage / Progress** 四頁;以下地方全部用「原始標單」計算:

| 檔案 | 影響 |
|---|---|
| `Payments.jsx:23` | 本期估驗/應領/保留款 與估驗頁不同 |
| `Dashboard.jsx:49` | 首頁完成率 vs 進度頁完成率不同 |
| `MonthlyReport.jsx:29` | 月報金額與估驗單不一致(對外文件!) |
| `RiskAudit.jsx:27` | 機關稽核用的是舊分母 |
| `SupervisorReport.jsx:33` | 監造報表同上 |
| `lib/assistantData.js:21` | AI 助理引用的數字與畫面矛盾(違反 assistant-chat 的一致性紀律,AI 還會照規則主動說「資料矛盾」) |

有核准追加減的案子一上線就會被抓到「每頁數字不一樣」。這正是 memory 裡「財務單一真相層」要防的事。
**修法**:把「調整後 items + 變更後契約金額」上收為 store 的 derived value(一處 useMemo),所有頁面吃同一份;禁止頁面自己 build tree。

### 🔴 B-03 契約義務「重新解析」先刪後插,失敗即毀損
`ledger.js parseContractBody`:先 `delete` 整表 `contract_obligations`,才 insert AI 新結果。AI 逾時/insert 失敗 → 舊清單已經沒了;AI 回 0 筆也會清空;已標「已提送」的完成狀態一律歸零。
**修法**:先 insert 成功再刪舊(或 RPC 包交易);至少把「已提送」狀態依 title 對映保留。

### 🟠 B-04 基準日(決標/開工)假成功
`Contract.jsx setAnchor` 忽略 `updateProjectAnchors` 的回傳 error。非專案建立者改基準日:RLS 靜默 0 列,本地 `anchors` state 卻立即顯示新值 → 整頁義務時程的到期/逾期/罰款試算都建立在一個沒存進 DB 的日期上,重整才還原。
**修法**:與座標儲存(SiteLog saveCoords)一樣檢查 error 並顯示「僅專案建立者可修改」。

### 🟠 B-05 saveSiteLog 先刪明細再插入,中途失敗掉資料
`site.js saveSiteLog`:upsert 表頭後 `delete daily_log_items` → insert 新列。insert 失敗時該日全部工項數量已被刪掉,回傳的 error 也只說 insert 失敗。估驗草擬吃日誌累計,等於估驗基礎資料默默蒸發。
**修法**:改 upsert-by-(log_id, work_item_id) + 刪除缺席列;或 RPC 交易。

### 🟠 B-06 escalateObservation 忽略缺失建立失敗
`collab.js:291`:`createDefect` 的 `{error}` 沒接,失敗時觀察事項照樣被標「轉缺失」→ 缺失不存在、觀察也關了,事項就此消失。另:markup 只有 `data:` URL 會帶過去,已存 Storage 的標註不會轉掛(次要)。

### 🟠 B-07 「DB 成功才更新 UI」原則的殘留破洞(RLS 靜默 0 列=假成功)
全案大多數寫入已改為 `mutationOutcome` DB-first,但以下仍是樂觀更新、不檢查、不回滾:

- `site.js deleteSiteLog`(刪日誌)/ `deleteSitePhoto`
- `ledger.js updateCostItem`(UI 先改,error 有回傳但呼叫端多未顯示)、`deleteCostItem`、`deleteChangeOrder`、`updateObligationStatus`
- `quality.js deleteInspectionPoint`
- `collab.js updateObservation`、`deleteObservation`
- `Quality.jsx submitInsp`:`createInspection` 的 error 直接吞掉,表單關閉如同成功

**修法**:統一走 `mutationOutcome` 模式 + 頁面 errMsg banner(Valuation/Payments 已有現成樣板)。

### 🟠 B-08 預定進度寫入 fire-and-forget
`billing.js generateSchedule`:async IIFE 先 `delete schedule_periods` 再 insert,**不 await、不檢查 error**——失敗時 UI 上 S 曲線好好的,DB 是空的或半套;`updatePlannedPct` 的 upsert 同樣 `.then(() => {})` 吞掉結果。S 曲線是機關端進度考核基準,不能假成功。

### 🟡 B-09 db.js 載入層全部忽略 error
所有 `loadXxxFromDB` 都是 `const { data } = await …`,查詢失敗=靜默空陣列。使用者看到「尚無缺失/尚無估驗」而非「載入失敗請重試」——標單有 error state + 重試 banner,其他 13 個領域沒有。對「以為缺失都結案了」這種誤讀,風險不小。

### 🟡 B-10 送審/RFI 編號用 length+1,刪除後重號
`SUB-${length+1}`:刪 SUB-002 後新建的又叫 SUB-003(和既有 SUB-003 撞號)。改用 max(既有編號)+1 或 DB sequence。

### 🟡 B-11 Dashboard 的 TODAY 是模組層常數
`Dashboard.jsx:13`、`todayISO` 在 module load 時算一次;工地平板整週不關分頁,日期、預定進度內插、落後判斷都停在開啟那天。Alerts 頁是每次 render 算(正確),兩頁行為不一致。

### 🟡 B-12 pdf.js worker 走 jsdelivr CDN
`db.js:276`、`documentExtract.js:100` 從 `cdn.jsdelivr.net` 載 worker。機關內網/斷網/CDN 被防火牆擋 → 契約 PDF 抽字直接掛,且錯誤訊息不會說是 CDN 問題。Vite 支援 `new URL('pdfjs-dist/build/pdf.worker.min.mjs', import.meta.url)` 打進自家 bundle。

### 🟡 B-13 auth 初始化失敗=全站永久「載入中…」
`auth.js`:`getSession()` reject 時 `authReady` 永遠 false,無 timeout、無重試 UI。

### ⚪ B-14 CSV 匯出無 formula injection 防護
`exportCsv` 對 `=`, `+`, `-`, `@` 開頭的儲存格(如工作摘要自由文字)未加 `'` 前綴,Excel 會當公式執行。機關收檔場景值得防。

### ⚪ B-15 送審附件換副檔名重傳留孤兒檔
`uploadSubmittalFile` 路徑 `<id>.<ext>`、`upsert:true`:先傳 .pdf 再傳 .docx,舊 .pdf 永留 Storage。

## 二、效能

### 🔴 P-01 Store context value 每次 render 全新物件,全站跟著重繪
`store.jsx:259` 的 `value` 是普通物件字面量,**沒有 useMemo**。任何一個 slice 的任何 setState(估驗打一格數字、聊天打一個字)→ StoreProvider re-render → value 參考變 → **所有** `useStore()` 消費者(30 頁中掛載的那頁 + Layout + WorkbenchTabs + CopilotFab + ConfirmHost)全部重渲染。大標單頁(估驗樹、BOQ)已經靠 onBlur 寫入避開打字卡頓,但根因還在。
**修法(遞進)**:① 先 `useMemo` 包 value;② 中期把 state 與 actions 拆兩個 context(actions 參考永遠穩定);③ 長期上 selector 式訂閱(use-context-selector 或 zustand)。

### 🟠 P-02 零 code splitting:30 頁全部打進首屏 bundle
`App.jsx` 靜態 import 全部頁面 → 主 chunk **1.18MB(gzip 362KB)**。登入頁也要先吞完估驗樹、SVG 圖表、Sentry、全部頁面。mammoth/pdfjs/範例標單已是動態 chunk(做得對),就差路由層。
**修法**:頁面全部改 `React.lazy` + `<Suspense>`(印表頁、Assistant、RiskAudit 這種低頻頁收益最大);預估首屏 JS 至少砍半。

### 🟠 P-03 切專案 = 15+ 次序列 round-trip
`store.jsx` dbMode effect 依序 await 10 個 loader(每個內部還有 1–2 個查詢)。行動網路 RTT 200ms 時,切案要 3–4 秒才齊。註解說「避免同時打爆連線」,但 supabase-js 是 HTTP/2 多工,並發 3–4 組完全安全。
**修法**:分組 `Promise.all`(估驗+進度+日誌 / 品質三表 / 成本+變更+排程),或乾脆仿 `portfolio_summary` 做一支 RPC 一次回。

### 🟡 P-04 Dashboard 的 `tallyBalls` 未 memo
`Dashboard.jsx:22` 每 render 重算(myOpenItems 有 memo、它沒有)。量小,順手修。

### ⚪ P-05 demo 範例標單 chunk 1.06MB
只在 sample 模式載入,真專案不吃;但 demo 銷售站首載會下載 gzip 129KB。可把 workItems.json 數值精度與欄位縮短,或 lazy 到進入 BOQ 頁才載。

## 三、UI/UX

### 🟠 U-01 唯讀角色的表單只做了一半
SiteLog 對監造/機關顯示「此頁唯讀」,但實際只有「天氣(上午)」有 `disabled`;**天氣(下午)、工作摘要、工項數量、公定格式全部欄位、「AI 批次辨識照片」「直接加照片」按鈕**都可以操作——輸入到最後才發現沒有存檔鈕。建議:`can.edit` 統一 disable(欄位加 `disabled={!can.edit}`、照片區塊整塊隱藏)。

### 🟠 U-02 打錯網址沒有 404,被靜默丟去首頁
`App.jsx` `*` 路由 → `/login` → 已登入又被彈回 dashboard/portfolio。使用者收藏的舊連結壞掉時,完全不知道發生什麼事。給一個「找不到此頁」畫面 + 回首頁連結。

### 🟡 U-03 錯誤回饋樣式三套並存
Valuation/Payments/Quality 有紅色 errMsg banner(好);SiteLog 用 savedMsg 小字(過目即忘);B-07 列的那批則完全沉默。建議抽一個 `useActionError` + 統一 banner/toast,所有寫入動作走同一條回饋路。

### 🟡 U-04 ProjectSwitcher 下拉的鍵盤/焦點可用性
無 `aria-expanded`、無 Escape 關閉、開啟後焦點不進選單;刪除專案與新增專案同一層級擠在切換器裡。requireText 確認設計得好,但入口位置值得再想(放專案設定頁更合理)。

### 🟡 U-05 index.html 幾乎裸奔
無 favicon、無 `<meta name="description">`、無 og tags。傳連結給機關承辦時,預覽只有網址。配合 B-01 一起整理 head 即可。

### 🟡 U-06 Alerts / Dashboard 行動中心用 index 當 key
`key={i}` 在常增刪的清單(提醒中心)會造成 DOM 錯配與動畫閃跳;items 都有穩定識別(title+to 或原始 id),改掉成本低。

### ⚪ U-07 主題只有亮/暗二態
無「跟隨系統」選項;首載有讀 `prefers-color-scheme`(對),但使用者手動切過一次就永遠釘死。

### ⚪ U-08 Payments 金流欄位無角色歸屬提示
DB 已有順序不變量(請款→收款→實收),但 UI 沒說「請款日=廠商填、收款/實收=誰填」;機關與廠商都能改同一欄,實務上容易互相蓋。至少在欄位標籤註明歸屬。

## 四、Workflow(產品流程)

### 🟠 W-01 義務時程「已提送」缺佐證鏈
契約義務只有一顆「標為已提送」toggle,沒有連到實際提送的送審(submittal)或文件。機關稽核視角(勾稽鏈是賣點)這裡斷了一環:建議義務可掛 submittal id 或文件 version id,「已提送」自動由關聯推導。

### 🟡 W-02 estimate 假成功風險集中在「非核心」領域
核心金流/品質已 DB-first,但成本管理、觀察事項、義務狀態這些「輕量」領域(B-07)恰好是廠商日常最高頻的操作;上線後最先被回報的往往是這種「我明明改了怎麼不見了」。建議當一批修。

### 🟡 W-03 切專案清 state 與載入的競態縫隙
清空 effect 與載入 effect 分開跑,依 React effect 順序目前正確(清空先註冊);但 `submittals/rfis/observations` 的載入在 `isPersistedProject` effect 內,與 dbMode effect 的缺失載入有一段時間差,切案瞬間仍可能短暫顯示混合狀態。可接受,但若要根治:把「清空+載入」合併為單一 effect,以 `projectId` 為 key 全量替換。

### ⚪ W-04 demo 與真實模式的行為差異未標示
demo 的檢查表判定/缺失連動在本地引擎,真專案下沉 DB trigger——兩邊規則若日後改版不同步,銷售 demo 展示的行為會與正式版不符。建議在 CLAUDE/開發文件標註「雙引擎需同步」清單(qc.js ↔ trigger、contractDue.js ↔ contractDue.ts 已有測試對齊,是好範本)。
